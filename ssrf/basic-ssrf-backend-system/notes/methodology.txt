# ============================================================
# METHODOLOGY — Back-End System SSRF Attack
# How to think through and execute this class of attack
# ============================================================


## PHASE 1 — RECONNAISSANCE: FIND THE SSRF ENTRY POINT

Goal: Find a parameter where the server fetches a URL you control.

What to look for in requests:
    stockApi=
    url=
    src=
    uri=
    path=
    dest=
    resource=
    feed=
    callback=
    endpoint=
    imageUrl=
    webhookUrl=

Features to test:
    - Stock / price check features
    - PDF or screenshot generators
    - "Import from URL" features
    - Webhook configuration
    - Social media link previews
    - XML / RSS feed subscribers
    - API integrations with callback fields

Confirm it works — change the value to your Burp Collaborator URL.
If you get a DNS or HTTP hit: confirmed SSRF.


## PHASE 2 — HOST DISCOVERY: FIND LIVE INTERNAL MACHINES

Goal: Identify which IPs in 192.168.0.X range have running services.

Method — Burp Intruder:
    1. Set stockApi=http://192.168.0.§1§:8080/
    2. Payload type: Numbers | From: 1 | To: 255 | Step: 1
    3. Run attack
    4. Sort by response LENGTH or STATUS CODE
    5. Outliers (different length or non-500 status) = live host

Reading results:
    HTTP 200          → service is running and responded normally
    HTTP 404          → host alive, path does not exist
    HTTP 302/301      → host alive, redirect (follow it)
    HTTP 500 + error  → most likely host NOT alive (app-level error from failed fetch)
    Timeout           → host dead or port filtered


## PHASE 3 — SERVICE ENUMERATION: WHAT IS RUNNING?

Goal: Understand what service is on the port you found.

Once you identify a live IP (e.g., 192.168.0.68):
    - Read the response body — services usually identify themselves
    - Check for HTML title tags, server headers, version strings
    - Try common paths: /admin /api /health /status /metrics /v1

Common internal services and their default ports:
    :8080  → Tomcat / Jenkins / Admin panels
    :9200  → ElasticSearch
    :6379  → Redis
    :5601  → Kibana
    :27017 → MongoDB
    :8888  → Jupyter Notebook
    :9090  → Prometheus
    :15672 → RabbitMQ management UI
    :8001  → Kubernetes API


## PHASE 4 — EXPLOITATION: USE WHAT YOU FOUND

Goal: Perform the actual attack action using the internal service.

For this lab — admin panel at :8080:
    Step 1: Confirm admin panel loads
        stockApi=http://192.168.0.68:8080/admin

    Step 2: Read the HTML — find available admin actions
        Look for links, forms, API endpoints in the response

    Step 3: Execute the admin action
        stockApi=http://192.168.0.68:8080/admin/delete?username=carlos

    Step 4: Verify
        HTTP 302 redirect back to /admin = success


## PHASE 5 — ESCALATION: WHERE TO GO NEXT

After confirming SSRF on a back-end system, escalate:

    → Scan all 255 IPs on multiple ports to build a full internal network map
    → Read /proc/net/arp via file:// to get ARP table (faster than scanning)
    → Try cloud metadata: http://169.254.169.254/latest/meta-data/
    → Probe discovered services for known vulnerabilities
    → Chain: Redis at :6379 → no auth → write cron job → RCE
    → Chain: ElasticSearch at :9200 → no auth → dump all data


## METHODOLOGY SUMMARY (COMMIT THIS TO MEMORY)

    1. Find the parameter that accepts a URL
    2. Confirm SSRF with Collaborator
    3. Scan 192.168.0.1–255 on the target port with Intruder
    4. Find the 200 — that is your live internal service
    5. Read the response — understand what you are looking at
    6. Execute your goal using the internal service's own API
    7. Escalate — every internal service you find is a new attack surface
